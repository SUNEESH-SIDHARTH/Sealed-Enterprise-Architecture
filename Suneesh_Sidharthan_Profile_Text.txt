SEVEN PRINCIPLES OF SEA
-----------------------

P1 — Contracts are durable; implementations are transient
     Contracts live in their own files, language-agnostic, forever.
     Implementations are free to change — new language, new framework,
     new AI-assisted rewrite.

P2 — Intent is captured, not just behaviour
     Why a contract exists is as important as what it does. Intent documents
     are append-only, preserving the full history of thought.

P3 — Succession is designed, not assumed
     Every sealed contract has a designated Owner. Succession is formal and
     traceable — not political.

P4 — Change is graduated, not binary
     Proposed → reviewed → parallel-live → migrated → adopted or retired.
     No silent merges, no shortcuts.

P5 — Purpose-change has its own protocol
     Material purpose shifts trigger a Major Version Bump — not a series of
     silent feature additions.

P6 — Portability is a first-class property
     Every durable artefact is expressed in YAML/Markdown — portable across
     organisations, stacks, and AI tools.

P7 — AI-readability is structural
     Contracts are designed to be read by future AI models without human
     intermediation.

The Seal Test: "If this element changes, does the system become a different
system?" Yes → durable (sealed). No → transient.

FIVE ARCHITECTURAL LAYERS
--------------------------

Layer 1 — Contract Definition
  Declare what must persist. Sealed contract in YAML. Language-agnostic.

Layer 2 — Intent Capture
  Capture why it must persist. Intent document in Markdown. Append-only.

Layer 3 — Verification
  Prove implementation conforms. Per-stack verifier. Contract is the
  architecture; verifier is enforcement.

Layer 4 — Governance
  Authorise change. Manage succession. Sealed Review Board. Quarterly cadence.

Layer 5 — Evolution
  Extend, major-version, or retire. Formal ceremony. Parallel-live replacement.

SEAL v3 — FIVE-RING GOVERNANCE FRAMEWORK
================================================================================

A production-validated implementation of SEA 1.0 principles. Deployed across
the MiVA platform — 84 Kotlin source files, multiple closed beta releases.

Authority Hierarchy:
  DATABASE > DECISION LOG > SSOT > Code > Console > Docs

Ring 1 — Self-Describing Headers
  Every sealed file carries a SEALED header with freeze date and human-readable
  enforcement warning. AI models and developers see the boundary immediately.

Ring 2 — Physical sealed/ Directory
  Bright-line boundary between frozen contracts and mutable extension layer.
  19 sealed API functions in SealedApi.kt — eternal contracts, never modified,
  never removed. New capability via named extensions in ExtendedApi only.

Ring 3 — Git Pre-Commit Hook
  Blocks any commit touching the sealed/ directory. Overrides require a
  documented audit trail entry.

Ring 4 — Build Hash Verification
  SHA256 hashes of all sealed files checked at build time. Any modification
  fails the build before reaching production.

Ring 5 — Runtime Integrity
  SealedContractVerifier runs at app startup, validating function count and
  names against the embedded manifest. Drift surfaces immediately.
